## Ключевая идея *(порядок на семинаре)*

1. **ИИ и doc** — как ускорить работу над ML System Design Doc: искать неопределённости, противоречия, слепые зоны и риски.
2. **Вопросы стейкхолдерам** — как находить вопросы заказчику, **ЧПР** *(человеку, принимающему решения)*, смежным подразделениям.
3. **Итерации с научруком** — структурирование знаний, протоколы встреч, обновление doc в git.
4. **Вишенка на торте** *(коротко, без live-кода)* — как после этого doc становится контрактом для генерации кода и роли **архитектора / Lead DS–PO**, а не джуна.

Хорошо проработанный ML System Design Doc — не бюрократия, а **контракт с ИИ-копилотом** (Cursor, Claude). Последствия:

- ускоряют проект позже;
- делают специалиста **архитектором**, а не заменяемым кодером;
- дают ИИ контекст для **review doc и генерации вопросов** — не для подмены мышления.

**Код к зачёту не обязателен** — см. [[Критерии зачёта — ML System Design Doc]]. На этом семинаре **не пишем и не генерируем модуль**.

## Маппинг ролей: научрук ↔ «бизнес» в шаблоне

| В шаблоне SDD | У магистранта |
|---------------|---------------|
| Product Owner / заказчик | **Научный руководитель** (или эксперт предметной области) |
| ЧПР / спонсор go/no-go | Научрук или представитель «бизнеса» на пилоте |
| Встреча с бизнесом | Консультация / созвон с научруком |
| Жёлтые вопросы в doc | То, что ещё не согласовали |
| Критерии пилота | Критерии успеха итерации / внедрения ВКР |

## Содержательная часть

### Блок 1. Мотивация: один проект — два горизонта (~7 мин)

- [ ] ВКР и ML System Design Doc — **один проект** — [[Слайд — ВКР и ML System Design Doc]]
- [ ] К зачёту обычно **≤3 встреч** со стейкхолдером; ноябрь и первая половина декабря = окно консультаций
- [ ] ИИ усиливает тех, кто умеет **формулировать**, а не тех, кто просит «напиши систему»

### Блок 2. ИИ и doc: пробелы, противоречия, слепые зоны *(приоритет 1, ~20 мин)*

- [ ] Live: открыть фрагмент демо-doc → промпт: *«Проверь §1 по [checklist](https://github.com/IrinaGoloshchapova/ml_system_design_doc_ru/blob/main/checklist.md). Найди пробелы, противоречия с §2 (если есть), слепые зоны. Предложи вопросы научруку. Не выдумывай факты.»*
- [ ] Показать: ИИ находит **дыры и несостыковки**, а не заменяет встречу
- [ ] Связь с реестром рисков [[08 ALL together#Модуль: риски этапа|лек. 8]] — риск как «мы этого ещё не знаем / не согласовали»

### Блок 3. Вопросы заказчику, ЧПР, смежникам *(приоритет 2, ~15 мин)*

- [ ] Handout: **«12 вопросов на 1-ю встречу»** *(из [[02 Business Understanding#Модуль: риски этапа|лек. 2]] и [[03 Data Understanding + DataSelection|лек. 3]])*
- [ ] Структура `script-before.md` — что спросить **до** встречи с учётом текущих пробелов в doc
- [ ] Практика: 3 **жёлтых** вопроса в своём doc
- [ ] Peer review: понятны ли вопросы соседу без устных пояснений?

### Блок 4. Демо-репозиторий: итерации и протоколы *(приоритет 3, ~20 мин)*

Показать **коммиты 0–3** *(итерации 4–6 в репо — тизер «следующий семестр», без live-кода)*:

| Итерация | После встречи | Что в doc | ИИ *(дома / на след. шаге)* |
|----------|---------------|-----------|------------------------------|
| 0 | — | Пустой шаблон | — |
| 1 | Первая беседа о теме | §1 черновик, жёлтые вопросы | Review §1 |
| 2 | Цели, TO-BE | §1 согласован, начало §2 | Review §1↔§2 |
| 3 | Данные, доступ, смежники | §2.3 этап 1 | Review таблицы данных |

```
sessions/NN/
  script-before.md
  protocol.md
  script-after.md    # «что забыл спросить»
```

- [ ] `git log` / diff doc — **не читать весь doc**
- [ ] Протокол и «ой, забыл спросить»
- [ ] Запись с Денисом Карамушко — опционально фрагмент; основной формат — **protocol.md**

### Блок 5. Вишенка: doc → архитектор, не джун *(приоритет 4, ~5 мин)*

- [ ] Тезис: после итераций doc достаточно плотный → копилот генерирует **скелет** pipeline / тесты / заглушки — **после зачёта**, не сегодня
- [ ] Роль студента: **Lead DS–PO** задаёт рамки; ИИ исполняет в рамках doc
- [ ] **Не делать** live-demo генерации модуля на семинаре

### Блок 6. Анонс ноября и домашка (~5 мин)

- [ ] [[10 Как заполнять Ч1–Ч2]] — §1 и §2 по шаблону
- [ ] [[11 Как заполнять Ч3–Ч4]] — §3, §4, roadmap
- [ ] ДЗ: **1-я встреча с научруком до ~10 ноября**, протокол в `sessions/01/`

### Блок 7. *(Только если осталось время)* Задачи на финализацию doc и риски

> Не обязательный блок. Не сдвигать блоки 1–6. Ориентир: **5–10 мин**.

- [ ] Промпт: *«По текущему doc и [[Критерии зачёта — ML System Design Doc|критериям зачёта]] составь **список задач** до сдачи: что дописать, с кем согласовать, что проверить. Отдельно — **риски** и пробелы по этапам CRISP-DM.»*
- [ ] Сверка с **чеклистом реестра рисков** [[08 ALL together#Модуль: риски этапа|лек. 8]]: Business → Data → … → Monitoring
- [ ] Итог для студента: черновик **backlog финализации** в репозитории (`tasks-finalization.md` или issues) — доработать к [[10 Как заполнять Ч1–Ч2|лек. 10]]

## Мемы для лекции

- **Копилот без doc** — «напиши мне ML-систему» → галлюцинация архитектуры.
- **Doc без встреч** — «мы додумали метрику сами» = красный флаг.
- **Git как дневник** — «протокол в коммите» vs «поговорили у кулера».
