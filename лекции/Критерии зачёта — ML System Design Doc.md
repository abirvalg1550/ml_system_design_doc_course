# Критерии зачёта — ML System Design Doc

> Официальные критерии **презентации и защиты ML System Design Doc с зачётом** по курсу ML Project Design.
> Тема doc = **тот же проект**, что и ВКР в магистратуре. Doc составляется **под внедрение**, а не «ради галочки».

См. также: [[Основные установки курса]] · [[Слайд — ВКР и ML System Design Doc]]

---

## Формат зачёта

| Элемент        | Описание                                                                                      |
| -------------- | --------------------------------------------------------------------------------------------- |
| **Что сдаём**  | ML System Design Doc (**§1–§4 обязательно**, в т.ч. деплой и мониторинг — см. минимальные форматы ниже) + **презентация** |
| **Как сдаём**  | **Презентация и защита ML System Design Doc с зачётом** — выступление + ответы на вопросы     |

| **Подготовка** | Лекции 6–8 (теория §4), семинары 9–11 (**§1–§3 + минималистичный §4**: Excel, batch, мониторинг «вручную» и т.д.), лекция 12 |

## Реалистичный контекст к моменту зачёта

К зачёту студент, как правило:

- только **познакомился с задачей** и начал её формализовать;
- успел **не более ~3 встреч** с заказчиком / стейкхолдером (научный руководитель, представитель «бизнеса», эксперт предметной области);
- **не обязан** успеть **production-деплой** или полноценный пилот в проде;
- **не обязан** иметь **рабочий код**, ноутбуки или обученную модель — **к зачёту достаточно doc**.

**Это нормально.** Зачёт оценивает **качество doc под внедрение**, **§4 (как доставляем и контролируем решение)** и **осмысленный план дальнейших шагов** — не факт промышленного запуска и **не наличие кода**.

> **Baseline, требования к данным (§2.3 этап 1), MVP** на зачёте — **описание в §2–§3** (методология, метрики, plan). **EDA и выгрузка к зачёту не требуются** — достаточно проработанных **требований к данным**. **Обязательный следующий шаг** — **следующий семестр** *(ориентир ~**3,5 мес после зачёта** курса)*: **доступ к данным** + **EDA выполнен**, §2.3 этап 1 обновлён **фактами**. Код и обучение модели — в roadmap **до 31 мая**.

## Обязательные требования к doc

### 1. Doc под внедрение вашего проекта

- [ ] §1–§4 заполнены **по вашей теме магистратуры**, не «шаблон на любой продукт».
- [ ] Бизнес-цель, AS-IS → TO-BE, критерии успеха пилота — **для реального внедрения / использования** (см. шаблон §1.2)
- [ ] Baseline **описан в doc** (§2–§3): с чем сравниваем, метрики ↔ бизнес — **код не обязателен**
- [ ] **§2.3 этап 1 — требования к данным:** цепочка **§2.2 → итоговые датасеты** (поля, **гранулярность** день/мин/сек) **→ сырые данные** (поля и гранулярность сырья) **→ доступ** (откуда, **должность · подразделение · продукт**, **как получить**); качество, FaO; **Data Selection** — train/val/test, репрезентативность *(жёлтым — несогласованное)*
- [ ] Жёлтые вопросы к «бизнесу» в doc — там, где ещё не согласовано ([workflow](https://github.com/IrinaGoloshchapova/ml_system_design_doc_ru/blob/main/ML_System_Design_Doc_Workflow.md)).

### 2. §4 Внедрение и мониторинг *(обязательно)*

§4 **обязателен** к зачёту. **Production не требуем** — достаточно **минималистичного**, но **конкретного** описания: как результат **доходит до пользователя** и как вы **контролируете**, что система не деградирует.

Разбор минимальных форматов — на **семинарах 9–11** (особенно [[11 Как заполнять Ч3|лекция 11]]); теория — [[06 Deployment]], [[07 Monitoring]], [[08 ALL together]]. Риски по этапам — [[Риски по этапам CRISP-DM — оглавление]] (лекции **2–8**).

#### Минимум по подразделам §4

| Подраздел | Обязательно | Что писать (минимум) |
|-----------|-------------|----------------------|
| **§4.1 Архитектура / интеграция** | **Да** | Схема или текст: **как** scoring/рекомендация попадает к пользователю; кто потребитель; batch / online / **Excel** / отчёт / API / встраивание в UI |
| **§4.3 SLA / нагрузка** | **Да** | Для MVP: частота обновления, допустимая задержка, объём — **хотя бы порядок величин** («раз в неделю batch», «ответ < 5 с») |
| **§4.6 Риски + мониторинг** | **Да** | Что мониторим (метрика модели, drift, бизнес-KPI); **как** (дашборд, ручная сверка, алерт, лог предсказаний); сценарий retraining / эскалации |
| §4.2 Инфраструктура | По ситуации | Для Excel/batch — «ноутбук + cron / ручной запуск» с обоснованием достаточно |
| §4.4–4.5 Безопасность | По ситуации | Если есть ПДн / compliance — обязательно; иначе — «не применимо» + почему |
| §4.6 Издержки | Желательно | Back-of-envelope для выбранного формата доставки |

#### Допустимые «минималистичные» форматы доставки *(не исчерпывающий список)*

- **Excel / CSV / Google Sheets** — выгрузка scoring, пользователь принимает решение в привычном инструменте
- **Batch-скрипт** — расчёт по расписанию, результат в витрину / файл / почту
- **Отчёт / дашборд** (BI, Streamlit, internal tool) — без «настоящего» ML-сервиса
- **Встраивание** — поле/кнопка в существующей системе, вызов API-черновика
- **Ручной пилот** — аналитик прогоняет модель, передаёт список рекомендаций заказчику *(с явным TO-BE в §1.2)*

#### Допустимые «минималистичные» форматы мониторинга

- **Ручная сверка** метрик раз в N (неделя / спринт) по чеклисту
- **Лог предсказаний** + периодический offline-анализ drift
- **Порог + алерт** (почта / мессенджер) при падении метрики
- **Сверка с бизнес-KPI** пилота по §3.2 — без Grafana в проде

> §4 **без** описания «кто и как получает результат» = **не сдан**. «Потом задеплоим в Kubernetes» без схемы MVP-пилота — **недостаточно**.

### 3. Риски *(обязательный блок)*

В doc **обязательно** выделить риски. Минимум — подраздел в **§4.6** (или §3, если риски пилота), плюс слайды на зачёте.

#### 3.1. Риски зависимости от **смежников** *(обязательно)*

Риски, где сильная зависимость **не от команды ML-проекта** (на курсе команда по сути — **один студент**), а от **смежных ролей / организаций**:

| Что указать | Пример |
|-------------|--------|
| **Кто смежник** | **должность · подразделение · продукт** *(не ФИО)* — пример: владелец данных · IT/DWH · *(продукт)*; PO · Закупки · … |
| **В чём риск** | нет доступа к данным, нет слота на деплой, нет согласования пилота |
| **План** | хотя бы **наивное**, но конкретное действие: к кому идти, что спросить, к какой дате, plan B |

> Не перечислять «мир изменился» без **должности · подразделения · продукта** со стороны смежника — так риск считается **не проработанным**.

#### 3.2. Честное разделение горизонтов

В doc и на зачёте явно разделить:

| Горизонт | Что писать |
|----------|------------|
| **К зачёту курса** *(этот семестр)* | Встречи с заказчиком, **doc §1–§4**, **§2.3 этап 1 — требования к данным** *(без EDA — нормально)*. **Кода может не быть** |
| **Следующий семестр** *(~**3,5 мес после зачёта**)* | **Доступ к данным** получен; **EDA выполнен**; §2.3 этап 1 **обновлён фактами**; бизнес **валидировал** агрегаты *(или риск + plan B задокументированы)* |
| **После EDA, до 31 мая** | Baseline/MVP в коде, пилот, §4 по факту — **roadmap** |
| **После магистратуры** | То, что **не успеете** к маю / к ВКР и почему |

## Риски и баллы: что штрафуем, что нет, что только в плюс

Не всё, что студент **не успел проговорить** с научным руководителем / заказчиком, одинаково влияет на оценку.

### Снижаются баллы

Если **можно было и нужно было** согласовать заранее, но в doc **нет ни ответа, ни жёлтого вопроса, ни риска**:

- бизнес-цель и **критерии успеха** итерации;
- **целевая переменная** и связь с бизнес-метрикой;
- **scope / non-goals** (что сознательно не делаем);
- **baseline** — с чем сравниваем MVP (**в doc**, не обязательно в коде);
- **§2.3 этап 1** — итоговые датасеты + сырые + гранулярность + **доступ** (откуда, **должность · подразделение · продукт**, как); train/val/test *(без привязки к §2.2 = пробел)*;
- формат **пилота** (как будем проверять успех);
- **§4.1** — как результат **доставляется** пользователю (не «abstract ML»);
- **§4.3 / мониторинг** — что и **как** контролируем на пилоте.

*Логика:* это зона ответственности студента как автора doc; «не успел спросить» без фиксации в doc = пробел.

### Не снижаются баллы

Если риск **объективно вне контроля** команды ML-проекта и **корректно описан**:

- зависимость от смежника с указанием **должности · подразделения · продукта**;
- **план** работы с риском (пусть наивный);
- явная пометка: «к зачёту не успели согласовать — ждём ответ от …»;
- **нет выгрузки / EDA** к зачёту — при **проработанных требованиях к данным** в §2.3 этап 1 и рисках доступа;
- **нет кода / ноутбука / обученной модели** к зачёту — при **полном описании** baseline и MVP в §2–§3.

*Логика:* не наказываем за то, что не зависит от студента, если doc это **честно отражает**.

### Только в плюс

- Уже есть **выгрузка / EDA / код / offline-оценка** — не требуется к зачёту, но уточняет §2.3 фактами;
- Продуманный блок **рисков смежников**: несколько зависимостей, **должность · подразделение · продукт**, plan B, связь с roadmap.
- Явное разделение **«к зачёту / до конца мая / после магистратуры»** с реалистичными сроками.
- Закрытые жёлтые вопросы там, где за ~3 встречи реально было что согласовать — с кратким протоколом «кто / когда / что решили».

## Roadmap: последующие шаги *(обязательно)*

В doc **и** в презентации на зачёте — таблица (или список) **последующих шагов**:

| Поле | Требование |
|------|------------|
| **Срок** | Шаги **до 31 мая**; обязательно — блок **«следующий семестр»** *(~3,5 мес после зачёта)*: **доступ к данным** + **EDA** |
| **DoD** | Definition of Done — **как понять, что шаг выполнен** (артеfact, согласование, метрика) |
| **Зависимости** | Если шаг блокируется смежником — ссылка на **риск** из §2.3 / §4.6 |
| **После мая** | Шаги после 31 мая — отдельно, с пометкой «после зачёта / после магистратуры» |

**Пример строк roadmap** *(после зачёта курса)*:

| Шаг | Срок | DoD | Зависимость |
|-----|------|-----|-------------|
| **Доступ к данным** (витрина / выгрузка) | **след. семестр**, ~3,5 мес после зачёта *(ориентир до 15.03)* | Выгрузка или sandbox; риск «нет доступа» закрыт или эскалирован | владелец данных · IT/DWH · …; DE · Data Platform · … |
| **EDA** + сверка требований §2.3 этап 1 | **след. семестр**, ~3,5 мес после зачёта *(ориентир до 31.03)* | Профилирование выполнено; FaO обновлены **фактами**; бизнес сверил агрегаты; §2.3 этап 1 без «жёлтых» по качеству | доступ к данным |
| Согласовать KPI пилота с заказчиком | до 15.04 | KPI в §1.2, жёлтое снято | заказчик / научный руководитель |
| Baseline offline (§2.3) | до 30.04 | Метрики baseline на hold-out; описание в doc | EDA |
| MVP-доставка (batch / Excel / отчёт) | до 15.05 | §4.1, пользователь получает scoring | см. §4.1 |
| Мониторинг пилота (минимум) | до 31.05 | §4.6: метрики, частота, plan retraining | — |

> К **зачёту** roadmap — **черновик**, но **не пустой**: видны шаги **«следующий семестр: доступ + EDA»** и дальше до 31 мая. **Без** этих строк roadmap **неполный**.

## Презентация на зачёте

Рекомендуемая структура (**5–10 мин** + вопросы):

1. **Бизнес** — кому помогаем, AS-IS → TO-BE, критерий успеха.
2. **Doc** — ключевые решения §1–§3 (не читать doc целиком).
3. **§4 — доставка и мониторинг** — как пользователь **получает** результат (Excel / batch / UI / …); что **мониторим** и как (минималистично — ок).
4. **Что уже сделано** — честно: встречи с заказчиком (~до 3), **содержание doc**. Кода может не быть.
5. **Roadmap** — **следующий семестр: доступ + EDA** (~3,5 мес); затем шаги **до 31 мая** *(baseline, MVP, §4)*.
6. **Риски смежников** — **должность · подразделение · продукт**, plan B.
7. **После мая / после магистратуры** — что за горизонтом курса.

**Типовые ошибки** (минус к оценке):

- много ML-техники, **нет** бизнес-метрики и baseline;
- doc **шаблонный**, можно перенести на другой продукт;
- **нет** §4 — как результат доходит до пользователя и как мониторим;
- **нет** roadmap с DoD до 31 мая **и без блока «следующий семестр: доступ + EDA»**;
- **нет** рисков смежников или «виноват мир», без **должности · подразделения · продукта** и планов.

## Rubric (ориентир для оценки)

| Критерий | Вес | На что смотрим |
|----------|-----|----------------|
| §1 Бизнес и критерии успеха | 15% | Цель, TO-BE, scope, согласованность / жёлтые вопросы |
| §2–§3 Методология и пилот | 20% | Baseline **в doc**, метрики ↔ бизнес, не шаблонность; **код не обязателен** |
| **§4 Доставка и мониторинг** | **15%** | §4.1 интеграция, §4.3 SLA (порядок величин), §4.6 мониторинг + риски; минималистичный формат — **ок** |
| **Риски и смежники** | 15% | Обязательный блок; штраф/плюс — см. таблицу выше |
| **Roadmap до 31 мая (DoD)** | 20% | **След. семестр:** доступ + EDA (~3,5 мес); далее шаги до мая; связь с рисками и §4 |
| Презентация и ответы | 15% | Структура, ясность, блок §4 на слайдах |

**Зачёт** — при выполнении обязательных требований и **≥ порога** по rubric *(порог уточняет преподаватель на консультации)*.
