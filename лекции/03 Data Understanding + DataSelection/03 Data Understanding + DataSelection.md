## Содержательная часть

### Data Understanding — требования к данным *(§2.3 этап 1)*

> **Логика для студента (к зачёту):** цепочка **§2.2 → итоговые датасеты → сырые данные → доступ**. EDA — **след. семестр** (~3,5 мес после зачёта).

- [ ] **Итоговые датасеты** *(target / features)* — **какие поля**, **гранулярность** *(день / час / мин / сек)*, ключи; связь с §2.2
- [ ] **Сырые данные** — что должно быть **в сырье**, чтобы собрать итоговые таблицы; **гранулярность сырья** vs итоговая *(агрегация, join)*
- [ ] **Откуда взять** — предположения *(FaO)*: система, витрина, выгрузка
- [ ] **Смежники для доступа** — **должность · подразделение · продукт** *(не ФИО)*; жёлтым — если не согласовано
- [ ] **Как получить доступ** — процесс: заявка, sandbox, NDA, встреча; **срок** в roadmap *(след. семестр)*; риск + plan B
- [ ] **Качество** — «чистота» итоговых данных; как DS проверит; **должность · подразделение · продукт** — кто от бизнеса валидирует агрегаты
- [ ] **FaO** — Facts / Assumptions / Opinions
- [ ] **Связь с §2.2** — каждая строка таблиц обоснована схемой baseline/MVP
- [ ] **Data shift** — риск для выборки и пилота

### Data Selection

> **К зачёту курса:** описать **требования к выборке и разбиению** под пилот из §1.2 — сущности, период, фильтры, **train / validation / test** *(минимальные объёмы, репрезентативность)*; **без** финальных сплитов и **без** EDA. **Следующий семестр** (~**3,5 мес после зачёта**): **доступ + EDA** — проверить по **факту**, что сплиты **реализуемы** и сабсемпл **представляет** целевую совокупность пилота; обновить §2.3. См. [[Критерии зачёта — ML System Design Doc|критерии]].

- [ ] **Отбор под пилот** — universe выборки = **будущий** TO-BE из §1.2, не «что удобно выгрузить»
- [ ] **Train / validation / test — требования к объёму** *(к зачёту — в doc, порядок величин)*:
  - **Train** — мин. число наблюдений / сущностей для обучения *(обосновать задачей)*; период, без leakage
  - **Validation / verification** — достаточно для подбора гиперпараметров / ранней остановки *(сколько фолдов / категорий / событий)*
  - **Test / hold-out** — достаточно для **стабильной** offline-метрики и guardrails *(min N, min events per slice)*; **не пересекается** с train по времени / сущностям, если так задуман дизайн
- [ ] **Репрезентативность сабсемпла** — как train/val/test должны **отражать** генеральную совокупность / пилот:
  - те же **сегменты** (регион, категория, канал), что в §1.2;
  - **стратификация** / квоты — если редкие классы / хвосты важны;
  - **временной срез** — train на прошлом, test на «как в пилоте» *(избежать optimistic bias)*;
  - **как проверим** после EDA: сравнение маргинальных распределений, агрегатов с бизнес-отчётом
- [ ] **К зачёту** — таблица требований к сплитам + **риски** (мало данных на test, selection bias) — [[#Модуль: риски этапа|риски этапа Data]]
- [ ] **След. семестр (~3,5 мес после зачёта)** — **EDA**: фактические размеры сплитов, проверка репрезентативности, §2.3 — **по факту**
- [ ] **Roadmap** — «доступ», «EDA + валидация сплитов и агрегатов с бизнесом» с DoD

## Мемы для лекции

- **Garbage In, Garbage Out** — без **требований к качеству** данные «есть», но MVP не реализуем.
- **FaO / жёлтые вопросы** — «данных нет» ≠ «§2.3 пустой»: описываем **что нужно** и **кого спросить**.
- **Data Selection без EDA на этом семестре** — к зачёту **требования к train/val/test** и **репрезентативности**; EDA — **след. семестр**, ~3,5 мес после зачёта.
